# Contributing to WorkOrder

Thank you for your interest in contributing to WorkOrder. Contributions help keep maintenance software practical and useful.

## Getting Started

Before contributing, please read:
- [Vision](docs/vision.md) - Understand what WorkOrder is and why it exists
- [Code of Conduct](CODE_OF_CONDUCT.md) - Our community standards

## How to Contribute

### Reporting Bugs

If you find a bug, please open an issue using the bug report template. Include:
- Clear description of the problem
- Steps to reproduce
- Expected vs actual behavior
- Relevant environment details

### Suggesting Features

Feature requests should align with WorkOrder's vision. Before suggesting:
- Check existing issues to avoid duplicates
- Consider if the feature makes maintenance clearer or more reliable
- Use the feature request template

### Code Contributions

1. **Fork the repository** and clone it locally
2. **Create a branch** for your changes
   ```bash
   git checkout -b feature/your-feature-name
   # or
   git checkout -b fix/your-bug-fix
   ```
3. **Make your changes** following our code style guidelines
4. **Test your changes** to ensure they work as expected
5. **Commit your changes** with clear, descriptive messages
6. **Push to your fork** and open a pull request

### Code Style

- Follow the project's `.editorconfig` settings
- Use clear, readable code that serves maintenance work
- Keep changes focused and practical
- Add comments where behavior might not be obvious

### Pull Request Process

1. Update documentation if needed
2. Ensure your code follows existing style patterns
3. Reference any related issues in your PR description
4. Be responsive to feedback and requested changes

## Questions?

If you have questions about contributing, feel free to open an issue for discussion.

Our goal is simple: make maintenance software that fades into the background so teams can focus on maintaining assets, not managing tools.
